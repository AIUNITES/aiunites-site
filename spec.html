Layer 1

Cross-Site Authentication

All AIUNITES sites share a single SQLite database (app.db) for user identity. A user who registers on any site in the network can authenticate on every other site with the same credentials. There is no OAuth, no third-party identity provider, and no token exchange — the sites share a database file that syncs through a Git-backed mechanism.

Registration follows a self-service model. The first user to register on any site is automatically assigned the admin role. Subsequent registrations create standard user accounts. Every site also supports a demo mode with pre-created demo users, allowing visitors to explore functionality without creating an account.

Sessions are managed per-site in localStorage. When a user logs in, their session state is stored locally and persists until explicit logout. There is no cross-site session cookie — each site independently verifies against the shared user table.

Admin capabilities include user management (view all users, usernames, join dates, avatars), system settings (email verification toggle, public signup toggle, max items limit, maintenance mode), statistics, and a versioned changelog.

Layer 2

CloudDB Sync

CloudDB is the data synchronization layer deployed across all 18 AIUNITES sites. It bridges the gap between localStorage (the primary data store for each site) and a cloud backend powered by Google Sheets via Google Forms and Apps Script. The architecture is offline-first: every site works fully without an internet connection, and syncs when connectivity is available.

Each site includes a cloud-database.js module that handles the sync lifecycle. Data flows from localStorage to Google Forms (as structured submissions), which land in Google Sheets. An Apps Script endpoint provides the read-back API for pulling cloud state into a local site.

The sync module exposes an online/offline toggle in the UI, letting users control when sync happens. When online, changes push automatically. When offline, changes accumulate locally and batch-sync on reconnection.

Sync Flow 
Write path:
  localStorage → cloud-database.js → Google Form POST → Google Sheet

Read path:
  Google Sheet → Apps Script API → cloud-database.js → localStorage

Conflict resolution:
  Last-write-wins with timestamp comparison

Layer 3

Webring Navigation

Every site in the AIUNITES network displays a fixed navigation bar at the top of every page. This bar lists all sites in the network, highlights the current site, and provides one-click navigation to any other site. It is the most visible expression of the middleware — the thing that makes the network feel like a network rather than a portfolio of unrelated projects.

The webring bar is maintained as a master template (webring-banner.html). When updated, changes propagate across all 18 sites. The bar uses inline CSS for zero-dependency rendering and includes a local development script that auto-converts absolute GitHub Pages URLs to relative paths when running on localhost or file:// protocol.

The current site is visually indicated with an underline and the primary brand color. The bar scrolls horizontally on narrow screens without hiding any sites.

Layer 4

MNN — Muscular Neuro Notation

Muscular Neuro Notation (MNN) is an open text format for describing human movement. It captures which muscles are engaged, the type of contraction, intensity, duration, and neural activation patterns. MNN is designed to be machine-parseable but human-readable — a movement can be described in a way that both a fitness app and a physical therapist can interpret.

MNN powers the BODWAVE movement layer in BodSpas (exercise and rehab logging) and the avatar posing system in InThisWorld (virtual world simulation). The same notation that describes a bicep curl can drive an avatar's arm position in a 3D environment.

The notation operates as a parameter space rather than a fixed vocabulary. Each movement is a set of key-value pairs describing body region, muscle group, action type, intensity (0–100), duration, and optional nerve pathway annotations. This means MNN can represent movements not yet catalogued — it is extensible by design.

MNN Record Example 
{
  "region":     "upper_arm",
  "muscle":     "biceps_brachii",
  "action":     "concentric",
  "intensity":  75,            // 0-100 scale
  "duration_ms": 2000,
  "nerve":      "musculocutaneous",
  "joint":      "elbow",
  "range_deg":  [20, 140],    // start, end angle
  "symmetry":   "bilateral"
}
ParameterTypeDescription
regionstringAnatomical body region (upper_arm, thorax, cervical, etc.)
musclestringSpecific muscle or muscle group (anatomical naming)
actionstringContraction type: concentric, eccentric, isometric, ballistic
intensitynumberEngagement level on 0–100 scale
duration_msnumberDuration of the movement phase in milliseconds
nervestringPrimary innervating nerve (optional)
jointstringJoint involved in the movement
range_degarrayStart and end joint angle in degrees
symmetrystringunilateral_left, unilateral_right, or bilateral

Consumer sites: BodSpas uses MNN for workout logging and nerve tracking. InThisWorld translates MNN into avatar joint rotations via Three.js. Gameatica references MNN structures in anatomy-related educational games.

Layer 5

VRN — Vocal Resonance Notation

Vocal Resonance Notation (VRN) is a parameter space for describing the physical production of vocal sound. It captures what the vocal apparatus is doing — larynx position, tongue shape, soft palate state, breath pressure, formant targets — rather than just the acoustic output.

The gap VRN fills: existing AI voice synthesis engines (ElevenLabs, OpenAI TTS, Google Cloud TTS) model voice as an acoustic waveform transformation. They can clone a voice or apply a style, but they have no representation of how the voice is physically produced. VRN provides that missing layer — a parameter space that maps to the physiological mechanics of vocalization. This makes VRN relevant both to vocal training (teaching a human to produce a sound) and to physically-informed AI voice synthesis (generating voice from a model of the vocal tract rather than purely from audio features).

VRN originated in COSMOS the Opera, where it was needed to notate specific vocal techniques for operatic performance. It was generalized into a standalone notation and is now the core protocol for VoiceStry's vocal training platform.

VRN Record Example 
{
  "larynx_height":   "neutral",     // low, neutral, high
  "vocal_fold":      "modal",       // breathy, modal, pressed, falsetto
  "soft_palate":     "raised",      // raised (oral), lowered (nasal)
  "tongue_body":     "mid_front",   // position in vowel space
  "tongue_tip":      "alveolar",    // contact point
  "jaw_opening":     45,            // degrees, 0 = closed
  "lip_rounding":    0.3,           // 0 = spread, 1 = fully rounded
  "breath_pressure": 60,            // subglottal pressure, 0-100
  "formants": {
    "F1": 500,  "F2": 1800,  "F3": 2600   // Hz targets
  },
  "pitch_hz":        262,           // fundamental frequency
  "duration_ms":     800,
  "vibrato": {
    "rate_hz": 5.5,  "extent_cents": 40
  }
}
ParameterTypeDescription
larynx_heightstringVertical larynx position: low, neutral, high
vocal_foldstringPhonation mode: breathy, modal, pressed, falsetto, fry
soft_palatestringVelopharyngeal state: raised (oral), lowered (nasal)
tongue_bodystringPosition in vowel space (high_front, mid_central, low_back, etc.)
tongue_tipstringArticulation contact: dental, alveolar, postalveolar, retroflex
jaw_openingnumberJaw angle in degrees (0 = closed, 60 = max open)
lip_roundingnumberLip shape on 0–1 scale (0 = spread, 1 = fully rounded)
breath_pressurenumberSubglottal pressure on 0–100 scale
formantsobjectTarget formant frequencies F1, F2, F3 in Hz
pitch_hznumberFundamental frequency in Hz
vibratoobjectVibrato rate (Hz) and extent (cents)
Why this matters for AI: Current voice synthesis treats the voice as a black-box waveform. VRN provides the missing parameter space for physically-informed synthesis — a model could generate speech by simulating vocal tract configuration rather than pattern-matching spectrograms. This is a fundamentally different approach and an open research direction.

DemoTemplate — Base Implementation

DemoTemplate is the canonical base for all AIUNITES sites. New sites are created by cloning DemoTemplate and customizing. It provides the full middleware stack out of the box: authentication, admin panel, user modals (settings, backup/restore, cache viewer, legal), CloudDB module, webring bar, dark theme, CSS variables, and responsive layout.

The template enforces AIUNITES conventions: vanilla JavaScript only (no frameworks), CSS custom properties for theming, [data-theme="dark"] for dark mode, semantic HTML5, and mobile-first responsive design. Navigation elements never use display: none — mobile menus use CSS transforms (slide-in patterns) to ensure accessibility and functionality.

Four sites currently run the full DemoTemplate feature set: VideoBate, AIByJob, Redomy, and DemoTemplate itself. One site (AIZines) has authentication but is missing the user modal layer. Ten sites are landing pages with CloudDB deployed but awaiting the full template upgrade. Three sites (COSMOS, AI YHWH, AIUNITES hub) are content-only by design.

Patterns

Admin Pattern Taxonomy

AIUNITES sites use one of three admin patterns depending on their complexity. Each pattern builds on the previous one.

📦

Cache Core

Simplest

localStorage and theming only. No authentication, no admin UI. Used by content sites and simple landing pages that just need theme persistence and data caching.

🔲

Modal Admin Core

Mid-tier

Authentication plus a modal-based admin panel. All admin functionality (users, settings, stats, changelog) lives in overlay modals. Used by games and single-page apps like VideoBate.

📄

Page Admin Core

Full-featured

Authentication plus dedicated admin.html and settings.html pages. For apps complex enough that admin functions need their own page context. Used by multi-page applications with extensive configuration.
